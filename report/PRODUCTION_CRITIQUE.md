# AegisRAG — Production-Readiness Critique

**Reviewer:** independent audit pass
**Scope:** full `AegisRAG/` tree, 99 Python files, ~15.5 K LOC
**Date:** 2026-04-17
**Bottom line up front:** the codebase is *architecturally* in the top quartile of graduate capstones and is ready to be demoed to a technical buyer, but it is **not** ready to be acquired, integrated, or put in front of paying customers. It's a working research prototype, not a product. Fixing that is mostly 6-10 eng-weeks of unglamorous platform work — not a re-architecture.

The sections below rank gaps by severity. "Acquirer" here means a mid-to-large SaaS company doing diligence; the questions are the ones their CTO, head of security, and SRE lead will actually ask.

## 1. Critical gaps (will block a deal)

### 1.1 Evaluation is not credible
The test set is generated by Qwen2.5-7B and then used to evaluate a Qwen2.5-7B-derived pipeline. Even with stratified doc-level splits, this is **self-evaluation** — the same model's distributional blindspots are baked into both sides. The FCRS numbers in the plan (0.84 on M5) are plausible in-distribution but **there is no out-of-distribution or human-labeled set** in `data/test/` at all.

- **Fix:** commission a held-out, human-labeled test set (500 queries, mixed domains, ≥2 annotators per item, Krippendorff α > 0.8). Budget: ~$3-5K via a labeling vendor, or 40 hours of internal SME time. Without this, every metric in the paper is suspect.
- **Related:** the BERTScore-derived "soft confidence labels" inherit the same generator's biases. Calibration can look great on in-distribution dev and still be badly off in production.

### 1.2 Zero committed training artifacts
`checkpoints/` is empty. No QLoRA adapter, no DPO adapter, no confidence-head weights, no alpha-network weights, no decomposer classifier. The pipeline tags `m1`–`m5` all *construct* fine-tuned components but there is nothing to load. **A reviewer running `python run.py query --model m5 ...` today gets only zero-shot base-model behavior.**

- **Fix:** run the full training sweep on a rented A100 (~$60-80, 12-14 hours), checkpoint to S3/HF Hub with content hashes, and add a `scripts/download_checkpoints.py` that fetches them on first use. Until this exists, claims like "B3 → M5 improves grounding from 0.80 to 0.93" are unverifiable.

### 1.3 No authentication, authorization, or rate limiting
`src/serving/app.py` ships CORS open to three hardcoded localhost origins and accepts any POST to `/query`. There is:
- no API-key or JWT middleware,
- no per-caller rate limit (one user can saturate the single generator),
- no quota/billing hook,
- no RBAC on `/tickets` (PII-heavy endpoint — anyone who can reach the server can dump every escalation with customer context).

- **Fix:** add a pluggable auth dependency (starlette-context or fastapi-users), rate-limit via `slowapi`, and put `/tickets` behind an admin scope. 1-2 engineering days.

### 1.4 PII is captured but never scrubbed
`AuditLogger` and `_SQLiteTicketStore` persist the raw `query`, `user_context`, and `evidence_gap` strings indefinitely. A customer typing their SSN or card number into the chat ends up on disk forever. For any regulated buyer (FinServ, healthcare, EU data subjects), this is an **immediate blocker**:

- No GDPR right-to-delete workflow.
- No PII detection / redaction before persistence (spaCy NER + presidio would be the standard answer).
- No data retention policy; audit.sqlite grows without bound.
- No encryption at rest for the SQLite file.

- **Fix:** add `src/serving/privacy.py` that runs presidio before `audit.log(...)`; configure retention TTLs; encrypt the DB file with SQLCipher; document a deletion API. This is 3-5 days and unblocks enterprise conversations.

### 1.5 Non-deterministic despite the claim
The plan promises "fixed seed, same query = same output." Reality:
- `torch.use_deterministic_algorithms(True)` is not set anywhere.
- `CUBLAS_WORKSPACE_CONFIG` is not exported.
- Reranker dropout isn't explicitly disabled at eval — relies on `model.eval()` being called; several code paths go through `__call__` without that guarantee.
- BGE-m3's MPS backend is not byte-identical to CUDA for the same weights.

Claim this is "near-deterministic" in the report, not deterministic, or actually lock it down with `src/utils/determinism.py` enforcing all the above.

## 2. Large gaps (will extend a deal timeline)

### 2.1 Test suite is cosmetic
10 test files, ~580 LOC — roughly 30-100 lines each. Most are import-and-instantiate smoke tests. What's missing:
- **No CGAL integration test** that walks a query through all 3 possible iteration paths (high-conf direct, mid-conf refine, low-conf escalate).
- **No streaming test** that exercises the SSE contract end-to-end with a fake client.
- **No fault-injection**: what happens when ChromaDB is down, when the reranker OOMs, when the NLI model fails to load?
- **No property-based tests** for citation parsing despite the regex being security-adjacent (malformed citations should never crash generation).
- **No CI**: `.github/workflows/` is absent. Tests only run when a human remembers to run them.

Target: 70% line coverage, plus one integration test per CGAL branch, plus a GitHub Actions workflow running on every PR.

### 2.2 Observability is a single-instance fiction
Prometheus counters in `/metrics` are fine as a toy. For production:
- No distributed tracing (OpenTelemetry would let you trace a single query across retriever → reranker → generator).
- Structured logs are mixed with loguru's pretty format; not JSON, so cannot be ingested by Datadog/Splunk/ELK as-is.
- No RED metrics (Rate, Errors, Duration) per component; current counters mix successes and failures.
- No SLO definitions. What is the p99 latency budget? What's the acceptable hallucination rate?
- Audit log is SQLite in the same container — loses data on restart if volume mount is wrong.

### 2.3 ChromaDB + in-process BM25 doesn't scale past one replica
`ChromaVectorStore` uses `PersistentClient(path=...)` — a local, file-backed collection. Two uvicorn replicas against the same volume is a data corruption risk (ChromaDB's locking is not designed for multi-writer). The BM25 index is a pickle file, reloaded per process. To scale:
- ChromaDB → Qdrant or Weaviate in client-server mode (or use ChromaDB's own server mode, added in 0.5+).
- BM25 → OpenSearch with BM25 plugin, or a dedicated service.
- Generator → vLLM server, not in-process HF Transformers.

Everything in `src/retrieval/` is written against a narrow interface, so this is a 1-week swap, not a rewrite. But it has to happen before you can accept real traffic.

### 2.4 Serving layer single points of failure
- A single model is loaded at startup; there is no blue/green deploy for model updates. Swapping the DPO adapter means bouncing the server and dropping in-flight streams.
- No graceful shutdown — `@app.on_event("shutdown")` closes the audit log but does not drain in-flight requests or abort SSE streams cleanly.
- No health check beyond "can I respond to GET" — `/health` returns 200 even when the downstream ChromaDB collection is empty.

### 2.5 No data lineage / model versioning
Every acquirer asks "which training run produced this checkpoint, and can I reproduce it?" There is:
- No MLflow / Weights & Biases / DVC.
- No dataset versioning (synthetic data is regenerated ad-hoc).
- No checkpoint hash in `QueryResponse` — you can't tell which model answered a given query from the audit log.

Minimum viable fix: MLflow local server (or Aim) and include `model_version_hash` in every response. 2-3 days.

## 3. Medium gaps (hygiene)

### 3.1 Supply chain
- `requirements.txt` is unpinned ranges; no lockfile.
- No hash-checking (`pip install --require-hashes`).
- No SBOM (`pip-audit`, `syft`).
- Dockerfile installs the entire build toolchain (`build-essential`, `git`, `curl`) and never removes it — final image is ~2-3 GB where it could be ~800 MB.
- Runs as root in the container.

Fix: switch to `uv` with a `uv.lock`, multi-stage Dockerfile, non-root user, `pip-audit` in CI.

### 3.2 Secret management
`.env.example` is the only convention. No integration with AWS Secrets Manager, Vault, or GCP Secret Manager. HF_TOKEN would sit in an env var on the machine.

### 3.3 Config sprawl
Three YAMLs (`base.yaml`, `lora_config.yaml`, `alpha_config.yaml`) plus `.env` plus hardcoded defaults sprinkled across `src/utils/config.py`. No schema validation at startup — a typo in `base.yaml` surfaces as an AttributeError deep in a pipeline.

Fix: pydantic-settings with a single `AegisConfig` root, `config.validate()` on startup, printed diff vs defaults.

### 3.4 Dead ergonomic corners
- `run.py` has 9 subcommands; no `--help` examples, no shell completion, no dry-run.
- Streamlit demo is a single `demo/app.py` with no auth, no session isolation, no transcript export.
- No CLI progress bars for long-running commands (ingest, generate-data, train).

### 3.5 Facade layer is honest but thin
`src/data/ingest.py`, `src/data/generate.py`, `src/training/train.py`, `src/evaluation/evaluate.py`, `src/serving/client.py`, `src/evaluation/benchmark.py`, `src/training/calibrate.py`, `src/decomposer/test_decomp.py` were added during this audit to satisfy `run.py` imports. They work, but they're CLI glue — if the CLI needs to grow (batch jobs, async tasks, parallel training), a proper job runner (Hydra / Prefect / Ray) should replace them.

## 4. What is genuinely strong

Being honest about the good parts:

1. **Clean layering.** `src/retrieval/`, `src/cgal/`, `src/tools/`, `src/serving/` are properly separated with narrow interfaces. You can replace any one module without touching the others.
2. **The CGAL design is real novelty**, not a rebranding. Confidence-gated AnswerVerify (skip for ≥ 0.85, async for 0.75-0.85) is the kind of engineering detail that shows up in production systems, not just papers.
3. **Six-type DPO is defensible.** The hallucinated-citation and wrong-tool rejection types are exactly the failure modes customer support LLMs exhibit in the wild.
4. **Adaptive alpha is measurably principled**, not just a hyperparameter rename. Per-query learned fusion is a genuine improvement over the fixed-alpha baseline.
5. **Type discipline.** Going from 111 pyright errors to 0 in an evening is not nothing; many "production" codebases have worse numbers.
6. **Determinism *intent* is present** in every module (seeded RNG, greedy decoding). The gap is enforcement, not architecture.
7. **Citation-weighted CE loss** is a clever, small detail that will actually move grounding numbers.

## 5. The 60-day plan to get acquisition-ready

Week 1-2 (security & integrity):
- Auth + rate limiting on FastAPI.
- PII scrubbing before audit persistence.
- Human-labeled held-out test set (500 queries).
- Actual training run → checkpoints committed with hashes.

Week 3-4 (observability & scale):
- OTEL tracing end-to-end.
- ChromaDB → Qdrant server mode.
- vLLM for generator serving.
- GitHub Actions CI (pyright, pytest, pip-audit).

Week 5-6 (hardening):
- 70% test coverage with integration tests.
- Multi-stage Docker, non-root, signed.
- MLflow for model versioning.
- Runbook + SLO doc.

Week 7-8 (polish):
- SOC 2 Type I prep (audit trail, access control docs).
- Load testing with Locust (100 QPS target).
- Customer-ready demo UI (not Streamlit).

After that, this stops being "a graduate capstone with good ideas" and becomes "a product a mid-size SaaS company would pay seven figures to acquire." Not because the ML got better — the ML is already the strongest part — but because everything around it finally matches the quality of the CGAL engine.
